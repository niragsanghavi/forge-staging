## 1. SCORING ENGINE STRESS TEST (`scoringEngine.js`, line-by-line)

**1.1 Month boundary — no crashes, one emotional landmine, one fairness gap (both known-by-design, one newly sharpened).**
- Perfect Week (`winStart<=DAYS-6`, lines 166-171): windows live strictly in 1..DAYS. A log on the 1st starts fresh; no undefined days, no NaN (DAYS≥28 always). A Jun-28→Jul-4 run earns nothing in either month — the documented cross-month exclusion, consistent with Invariant 6. Greedy left-scan with +7 jump is provably count-optimal for this structure. ✅ mathematically; the fairness gap stays parked (batch-2 consolidation).
- Team Streak: `qualByTeam` builds only from current-month logs; day-1 runs start at 1. No carry, no crash. ✅
- **The landmine (🟠 T-10 adjacent, product not code):** individual streak on the 1st: `checkUpTo = days.has(1) ? 1 : 0` → loop exits immediately → every user's flame reads 0/1 on the first of the month regardless of a 20-day June run. The engine is correct per spec; the SPEC is the month-boundary churn moment the pilot's Day-30 signal watches. No code change now — but this line is where "月2 start" lives or dies, and any future continuity feature (amnesty bridging is already contract-ready for a different case) lands exactly here.
- `todayDay = todayInSeason ? getDate() : DAYS` with a FUTURE season (clock-behind client, pre-created season): seasonPast=false, todayInSeason=false → todayDay=DAYS, zero logs → all-zero scores, no crash. ✅

**1.2 The WeakMap cache — the cache is sound; the INPUT PIPELINE is the hole.**
Attacked every invalidation path: twist toggle → twists snapshot builds a NEW `activeTwists` object → identity check `hit.twists!==twists` invalidates ✅. Same for `tw` (twistWindows), `bonuses`, `jacks`, `ips`, and `allLogs` (arrays replaced per snapshot, never mutated — the WeakMap key rotation is the design and it holds) ✅. `saveEditGroup` mutates `window.season` IN PLACE via `Object.assign` — `hit.cfg===cfg` passes, but every mutated scalar is in the `stamp` string, so invalidation still fires. Lucky-but-correct; fragile against FUTURE in-place fields — STRATEGY_PACK §3.2 already mandates adding the amnesty signature to `stamp`; make that a standing rule: **any new season field the engine reads must join the stamp** (recorded as T-12 guard note).
- 🔴 **T-2 — the real finding: there is NO season-document listener in `subscribe()`.** The cache can't invalidate on data the client never receives. Concretely: a new player registers, or an admin re-teams/re-roles someone, or super-admin edits targets — every OTHER open device keeps the stale roster/config until manual reload. Downstream: the new joiner logs workouts that appear in the feed (team falls back to `l.team`) but they are **invisible on the leaderboard** (renderLB maps `season.roster`) and excluded from team-streak maps on stale devices. The Batch-1 group-doc listener catches rollovers only. Fix: add a season-doc `onSnapshot` to `subscribe()` (registered in `unsub`, refresh `me` from the fresh roster exactly as `loadGroup` does, then `refresh()`). Cache invalidation follows automatically via the new roster reference.
- 🟡 **T-10 — midnight staleness:** `stamp` includes `toDateString()`, but re-computation only happens on a render, and renders only fire on snapshots/tab-switches. An app left open across midnight (or across month-end 11:59) shows yesterday's today-highlight, streak, and `isEnd` state until any data arrives. Fix: `visibilitychange` + a date-change check that calls `refresh()`.

**1.3 Iron Pledge settlement — no 11:59 PM loophole; a real backdating loophole instead.**
- Logging Sunday 23:59 local writes `day = Sunday` → counts toward that week. Legitimate, not a leak; settlement (post-Batch-1) refuses to run until `today > week's Sunday`. ✅ per Invariant 6.
- 🟠 **T-5 — the backdating window:** logs store user-chosen `day` integers, and the day strip legitimately offers the past 7 days. Sequence: pledge week ends Sunday short of target → Monday, BEFORE settlement, player backlogs "Saturday's workout" → settlement counts it (filter is day-range only). S2's auto-settle shrinks the window to roughly one app-open, but the cheater IS the first app-open. Options (decision, not code): (a) settlement counts only logs whose `timestamp` precedes week-end + 24h grace — closes the cheat but penalizes genuinely-offline Sunday logs syncing Monday; (b) accept, rely on the feed's existing "backlog" badge + flag culture (the pledge is social stakes among friends). RECOMMENDATION: (b) now, (a) revisited if the SME tier ships (money changes the threat model — MONETIZATION lane note added).
- 🔴 **T-1 — found while tracing the same path, and it's worse than the pledge: `pickDayFromInput()` performs ZERO validation.** `selDay = new Date(v).getDate()` from the "Earlier date?" `<input type=date>` — no month check, no year check, no future check (the strip validates; the picker doesn't). Consequences today: pick ANY month's 15th → logs day 15 of the CURRENT season; pick July 30 on July 9 → a FUTURE log — pre-log the rest of the month, pre-complete pledges, fabricate streaks and perfect weeks. This is the single largest integrity hole in the live app. Fix: clamp — parsed date must satisfy same month+year as season AND ≤ today (and the input gets min/max attributes for honest UX). Small, pre-pilot, ships as a bug fix under the freeze rules.
- 🟡 **T-7 — comedy bug, real points:** `bonus_workout` matches by SUBSTRING (`w.toLowerCase().includes(bonusWord)`). Bonus workout "Run" → "Brunch" earns +6/day. Fix: exact match on the normalized string (or word-boundary), one line.

## 3. ASYNC / RACE CONDITION HUNT (`appState.js`, `firebase.js`, submit path)
- **Double-tap on Log Workout:** SAFE. `btn.disabled=true` executes synchronously before the first handler yields; JS single-threading means the second tap sees a disabled button. ✅
- 🟠 **T-3 — the adjacent real bug: post-commit exception re-arms the button.** `submitLog`'s catch re-enables on ANY throw — including throws AFTER `batch.commit()` succeeded (Jack check, DOM ops, celebration). User sees "Error saving", taps again → **genuine duplicate logs** (the batch landed). Fix: after the commit resolves, nothing may route to the re-enabling catch — wrap the post-commit tail in its own try that logs but never re-enables (a failed celebration is not a failed save).
- 🟠 **T-4 — offline submit exact failure mode (with persistence, as shipped):** offline `batch.commit()` applies locally + queues, but the PROMISE stays pending until server ack. Sequence the user experiences: tap → "Saving..." forever → the log VISIBLY APPEARS in the feed (local listener fires with pending writes) while the button stays stuck → minutes later on reconnect, the success screen erupts out of nowhere. Secondary: the multi-group pre-reads (`get()` on other groups' docs) reject offline when uncached → correctly skipped per-session (only the active group receives the log — acceptable). Fix direction (PERFORMANCE_SHIELD §3 lane): treat sustained pending as optimistic success — a "Saved on this phone — will sync" state after a short timeout, never a hang. Decision needed on copy/threshold; the mechanics are a race between `commit` and a timer.
- 🟡 **T-11 — cosmetic:** pending-write logs have null `serverTimestamp` → feed sort (`timestamp?.seconds||0`) drops them to the bottom until ack. With persistence this is now user-visible. Fallback: sort key `timestamp?.seconds || Date.now()/1000` for own pending docs.
- `ensureAuth()` race matrix (listener + signInAnonymously, staging/prod split, double-settle of the promise): attacked, holds — a late rejection after a listener-resolve is a settled-promise no-op; the offline-first-launch path lands in the documented "Could not connect" screen. ✅ · localStorage session ops across tabs: last-write-wins, worst case a re-pick of group — benign. ✅ · `switchGroup`'s null-season window + Batch-1 rollover listener: teardown order verified (unsub before state swap). ✅

**Verdict, ruthlessly:** the engine's math is solid — perfect-week greedy, tie handling, cache identity discipline, and transaction ordering all survived attack. The app breaks not under load but under TIME and DISTANCE: unvalidated time input (T-1), state that never travels between devices (T-2), and promises that outlive the network (T-3/T-4). Two of those (T-1, T-2) are pre-pilot blockers by this auditor's judgment: one lets a pilot member fabricate a month, the other makes a pilot group's newest member invisible — both are exactly the kind of "product feels broken" signal that would contaminate the only experiment that matters.
