## A. WHAT FORGE IS (three sentences of orientation)
A social accountability fitness platform for friend groups — monthly seasons, daily workout logs, mutual witness (reactions, feed, shared goals), light competition on top. Vanilla JS + Firebase (Firestore, anonymous→custom-token auth) + GitHub Pages; one file of app logic (`index.html`), one solo founder-operator. The strategic bet (FORGE_NORTHSTAR.md): **the group is the atomic unit and the product is a self-running ritual for it** — every build decision is scored against a stranger-group completing a season with zero founder involvement.

## C. GLOBAL ARCHITECTURE INVARIANTS (no model may alter these — ever)
**Data & transactions**
1. Firestore transactions: ALL reads before ANY writes. No exceptions, no "just one quick read."
2. Idempotence = deterministic doc IDs + create-only rules (`{player}_{week}` bets, `{logId}_{player}` reactions, `{groupCode}_{player}_{week}` settlements, `payments/{payment_id}`). Client-side "did I do this" flags are never the guard, only cosmetics.
3. Roster arrays are rewritten ONLY inside `db.runTransaction` on a fresh read. Offline roster ops fail fast with a toast — a plain-`update()` fallback reintroduces the clobber race and is forbidden.
4. Logs are never deleted or content-edited: tombstones only (`voided/voidedBy/voidedAt`, plus write-once `photoPath`). Consumers filter voided at ingestion (one place: the listener).
5. Archived seasons (`status=='archived'`) are immutable — enforced in rules once AUTH_UPGRADE lands; treated as immutable by all code regardless. Snapshot ALWAYS precedes archive (a frozen season with no snapshot is unwritable forever).
6. Never settle, snapshot, or score a time period that hasn't fully ended in local time. Cross-month Mon–Sun weeks are excluded from every weekly mechanic (reuse the existing guard; do not re-derive).
7. Points are DERIVED from month-scoped logs — never stored, never "reset." Any code that writes a points value to a player record is corruption.
**Identity & security**
8. `userId` (app-level) is identity; `request.auth.uid` is a device until custom tokens land, after which they converge BY CONSTRUCTION of the token — never by assumption. The roster `uid` field is historical device data.
9. Scoring stays keyed by player NAME (`score(name)`, logsByPlayer, roster lookups). `userId` is additive; re-keying the engine is out of scope for any task, always.
10. No plaintext PIN is ever written anywhere (roster legacy `pin` is scheduled scrub debt). Admin/user verification compares hashes (`sha256hex` client; `node:crypto` + pepper server-side per AUTH_UPGRADE). No secrets, keys, or plaintext credentials in any committed file or commit message.
11. Every user-supplied string entering innerHTML passes `esc()`; no user data is interpolated into inline event handlers (data-attributes + listeners).
12. The client NEVER writes entitlements (`tier`, `proSeasons`) or payment records — webhook function only.
13. Public funnel surfaces (create-flow CTAs, landing) ship to prod ONLY in/after the same promotion as the security gate they depend on (NORTHSTAR §5.3). Funnel visibility and gate deployment are one commit, structurally.
**Performance & cost**
14. No full-collection reads on any recurring or user-facing path (`logs.get()` etc.). Month+group equality-scoped queries only; new queries stay equality-only (a composite-index requirement means STOP and surface — indexes can't deploy silently). The schema-v3 export is the sole sanctioned full scan (it is a backup).
15. Cross-group surfaces read pre-computed aggregates (PERFORMANCE_SHIELD), never fan-out scans. In-group scoring stays client-side from listener data — do not "optimize" it server-side; offline correctness depends on it.
16. Ambient/analytics writes are fire-and-forget behind try/logErr, never blocking a user action. Every listener registers in `unsub`.
**Process**
17. Staging first, always: `~/Desktop/forge-staging` (Firebase project forge-staging-865ff) → verify → promote file-copies to `~/Desktop/forge` (forge-25c8c). Rules deploy staging-first via CLI; prod rules deploy is a named checklist line. Real users live on prod: NO untested deploys.
18. Bump `CACHE_VERSION` in sw.js with any JS-behavior promotion. Never commit or push unless the task says to; keep untracked design docs out of commits.
19. Design system: Forged Glass tokens from `src/style/main.css` only — system font stack (ratified; Google Fonts retired), sparks/confetti are DOM patterns (never canvas), reduced-motion honored on all new motion, dark-first (updates/landing are dark-by-design).
20. **Do-not-build list (cite NORTHSTAR §0 and halt):** coins/rewards economy, gender/demographics collection, corporate-wellness product features, push notifications, Global-tab investment, heatmap UI, `users/*/activity` subcollections, cross-group streaks (until the Apr-2027 lane re-opens it), Home feed-first reorg (until the §7.3 pilot tripwire fires), pilot dashboards, expiry crons.

## E. STANDING ORDERS FOR AI AGENTS (hard rules of engagement)
1. **Read order for any task:** this file → FORGE_NORTHSTAR.md → the specific contract for your domain → the LIVE source files you'll touch (re-read them; grep for anchors; assume every line number in any doc has drifted).
2. **Halt protocol:** if a requested or self-generated change violates any §C invariant, contradicts the precedence chain, or requires something on the do-not-build list — **STOP. Implement nothing, including "temporary" versions.** Output a structured violation report: `⛔ INVARIANT CONFLICT — invariant #N / contract §X · proposed change · exact conflict · options with trade-offs` — and wait. Workarounds that technically skirt an invariant's wording while defeating its purpose are violations.
3. **Structural-mismatch halt:** if a contract references code/fields/rules that don't exist in the live repo (or vice versa), do not improvise the missing piece into existence and do not delete the "extra" — report the drift with file/line evidence and wait.
4. **Ambiguity rule:** contract silent + NORTHSTAR silent ⇒ ask. A guessed decision that ships is worse than a delayed one — this codebase's history proves interventions are cheap on staging and expensive after promotion.
5. **Scope discipline:** touch only what the task names. No drive-by refactors, no modernizing adjacent handlers, no "while I'm here." Batch-1's record — nine surgical commits, each independently revertable — is the required shape of work.
6. **Verification duties:** every change is verified on staging (preview + console + the relevant checklist) before it is called done; report failures verbatim, never soften. Pilot window (Aug 15–Sep 30, 2026): bug fixes ship, features freeze, pilot groups are never touched by hand.
7. **When in doubt about WHO decides:** mechanics → these contracts; product intent → NORTHSTAR; anything about money, security posture, the pilot, or deleting data → Nirag, explicitly, every time.
